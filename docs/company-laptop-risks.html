2. 위험 항목별 상세 분석

1

HIGH
▼

문제 근거

$ codesign -dv ClaudeNotifier
Signature=adhoc
TeamIdentifier=not set
Info.plist=not bound
  • 빌드 후 swift build -c release로 생성된 바이너리는 ad-hoc 서명만 가짐
  • Apple Developer ID / Team ID 없음 → Gatekeeper "확인되지 않은 개발자" 경고
  • 기업 MDM이 com.apple.systempolicy.control payload로 정책을 "App Store + Identified Developers"로 잠그면 수동 우회도 MDM 재적용으로 롤백됨
  • install-launchagent.sh:49 — launchctl load 시점에 quarantine 플래그 검사 → 서명 미통과 시 LaunchAgent 로드 실패

완화 방안

  • 회사 Apple Developer 계정으로 Developer ID Application 서명 + notarytool 공증
  • Jamf Self Service를 통해 패키지(.pkg)로 배포하는 방법도 유효
  • 단기: xattr -d com.apple.quarantine <binary> (MDM이 잠겨 있지 않은 경우만 작동)
2
Privacy Permissions — osascript / AppleEvents MDM 사전 거부 notify-hook.sh가 osascript로 Terminal TTY를 조회 → PPPC payload로 차단 시 터미널 포커스 감지 완전 불능
HIGH
▼

문제 근거

notify-hook.sh:110
frontmost="$(osascript -e \
  'tell application "System Events"
   to return name of first application process
   whose frontmost is true')"
notify-hook.sh:117
frontmost_tty="$(osascript -e \
  'tell application "Terminal"
   to return tty of selected tab of front window')"
  • macOS Mojave 이후 Apple Events는 개별 앱 단위로 Automation 권한 필요
  • MDM의 com.apple.TCC.configuration-profile-policy payload로 AppleEvents를 Deny로 잠글 수 있음
  • 차단 시: osascript가 -1743 (Not Authorized to Send Apple Events) 반환 → is_terminal_focused()가 항상 return 1
  • 결과: 터미널이 실제로 포커스되어 있어도 매번 배너가 팝업되는 오작동 또는, 반대로 모든 hook이 fail-open(무조건 허용)될 수 있음

완화 방안

  • MDM 관리자에게 PPPC payload에 ClaudeNotifier 및 bash를 AppleEvents → Terminal.app / System Events Allowed로 추가 요청
  • AppleEvents 없이 동작하는 폴백 경로 구현 (NSWorkspace API만 사용)
  • osascript 실패를 graceful하게 처리해 false-positive 배너 억제 필요
3
EDR 행동 탐지 — Hook Script 실행 패턴 bash가 ps + osascript + curl(localhost) 조합을 매 도구 호출마다 실행 → CrowdStrike/Falcon Behavioral Prevention 시그니처 매칭 가능
HIGH
▼

문제 근거

notify-hook.sh:192-202 — 프로세스 트리 순회
for _i in 1..10; do
  ps -p "$_tp" -o args= # 각 부모 PID 확인
done

notify-hook.sh:110 — osascript 호출
notify-hook.sh:185 — curl http://127.0.0.1:47823/event
  • 단일 bash 프로세스가 (1) 프로세스 열거, (2) Apple Events, (3) 로컬 HTTP 연결을 순차 실행 — EDR 행동 시그니처 조합 해당
  • Claude Code가 도구를 호출할 때마다(초당 수회 가능) hook이 반복 실행됨
  • CrowdStrike Falcon Behavioral Prevention: "Script executing network connection" + "Process discovery via ps" = MEDIUM 위협
  • 차단 시: hook 프로세스에 SIGSTOP 또는 종료 → notify-hook.sh가 응답 대기(최대 45초) 후 timeout → Claude Code가 45초간 stall

완화 방안

  • EDR 정책 담당자에게 notify-hook.sh 절대 경로를 프로세스 exclusion으로 등록
  • ClaudeNotifier 바이너리도 exclusion 목록에 추가
  • ps 프로세스 트리 순회 횟수 축소 (현재 최대 10회, 대부분 2~3회면 충분)
  • hook 타임아웃(CLAUDE_NOTIFIER_WAIT, 기본 45초) 축소로 stall 시간 단축
4
SSL Inspection Proxy — api.anthropic.com 차단 Zscaler / Cisco Umbrella가 AI 서비스 카테고리 차단 정책을 적용하면 Claude Code 자체가 API 호출 불가 (notifier의 간접 영향)
HIGH
▼

문제 근거

  • claude-notifier 자체는 외부 API를 호출하지 않음 (HTTPServer.swift:29 — 127.0.0.1만 바인드)
  • 그러나 Claude Code가 api.anthropic.com으로 HTTPS 요청을 보낼 때 기업 프록시의 SSL inspection을 통과해야 함
  • Zscaler의 "AI/ML Applications" 카테고리 차단 정책이 활성화된 경우 → api.anthropic.com 403 또는 SSL 오류
  • curl이 Homebrew 빌드(시스템 keychain 미사용)인 경우 기업 CA 인증서를 신뢰하지 않아 SSL 검증 실패 가능

완화 방안

  • IT 보안팀에 api.anthropic.com URL policy exception 요청
  • 기업 CA가 시스템 keychain에 설치된 경우 macOS 기본 curl은 자동 신뢰 (Homebrew curl 사용 시 --cacert 설정 필요)
  • 회사 프록시 설정이 있으면 Claude Code의 HTTPS_PROXY 환경변수 설정
5
비표준 포트 47823 TCP Listen localhost 전용이지만 비표준 포트 listen은 EDR "unusual service listening" 이벤트를 생성할 수 있음
MEDIUM
▼

문제 근거

  • 포트 충돌: 다른 앱이 47823을 사용 중이면 bindFailed(errno) → exit(2)로 앱 종료
  • Palo Alto Cortex XDR: 비표준 포트 listen → "suspicious service listening" 이벤트 생성 → 보안팀 조사 유발
  • 외부 노출 없음 (127.0.0.1 전용)이지만 같은 머신의 다른 프로세스가 인증 없이 접근 가능 (아래 항목 8 참조)

완화 방안

  • 환경변수 CLAUDE_NOTIFIER_PORT로 포트 변경 가능 (LaunchAgent plist에 EnvironmentVariables 추가)
  • 포트 충돌 시 다른 포트로 자동 retry 로직 추가 검토
  • EDR allowlist에 ClaudeNotifier 바이너리 등록으로 이벤트 억제
6
LaunchAgent — Santa / Jamf Protect 바이너리 차단 Google Santa 또는 Jamf Protect의 binary allowlist가 적용된 경우, 서명 없는 바이너리는 LaunchAgent 등록은 되지만 실행 즉시 SIGKILL
MEDIUM
▼

문제 근거

  • Jamf Protect "Restricted Software": 프로세스 이름 ClaudeNotifier 기준으로 실행 차단 설정 가능

  • Santa allowlist에 바이너리 SHA-256 hash 또는 TeamID 등록
  • KeepAlive를 false로 변경하거나 ThrottleInterval 증가 (반복 차단 알림 방지)

  • POST /clear에 인증 요구 추가 (현재는 완전 오픈)

4. 조치 권고사항

🔴 즉시 (코드 변경 없이 가능)

  • EDR exclusion 요청 — notify-hook.sh 절대 경로 + ClaudeNotifier 바이너리를 IT 보안팀에 프로세스 exclusion 등록 요청
  • quarantine 제거 — xattr -d com.apple.quarantine <binary> (MDM 미잠금 시)
  • jq 설치 확인 — brew install jq 또는 IT 채널로 설치 승인
  • CLAUDE_NOTIFIER_WAIT 축소 — 기본 45초 → 15초로 환경변수 설정 (stall 시간 단축)

🟡 단기 (코드 수정)

  • Developer ID 서명 — 회사 Apple Developer 계정으로 서명 + notarytool 공증 (install-launchagent.sh에 codesign 단계 추가)
  • 로그 내용 축소 — TOOL_SUMMARY를 도구명 + 길이만 기록하도록 notify-hook.sh:266-272 수정
  • jq 의존성 체크 — hook 상단에 command -v jq 확인 추가
  • KeepAlive 조건부 — Santa 차단 시 무한 재시작 방지를 위해 ThrottleInterval 값 증가

🔵 중기 (설계 개선)

  • Unix domain socket 전환 — TCP 포트 제거 → 파일 권한 기반 접근 제어 + EDR 포트 탐지 원천 차단
  • HMAC 인증 — 앱 기동 시 shared secret 생성, hook과 공유하여 HTTP 요청 서명 검증
  • os_log 전환 — Logger.swift를 macOS Unified Logging으로 교체 (자동 로테이션 + 암호화)
  • MDM PPPC payload 배포 — MDM 관리자와 협의하여 bash의 AppleEvents → Terminal.app Allowed 엔트리 등록
주의 — 항목 1(Gatekeeper)과 항목 2(Privacy MDM)는 개인 개발자가 독립적으로 해결하기 어렵고 회사 IT/보안팀 협력이 필수다. 먼저 보안팀에 용도(개인 생산성 도구, localhost 전용, 외부 API 미호출)를 설명하고 allowlist 등록을 요청하는 것이 가장 효과적인 첫 단계다.
